## 1. Objectif

Ce document définit l'architecture générale de l'API REST de TurfIA.

L'API constitue l'unique point d'accès aux traitements métier. Aucune
interface cliente ne dialogue directement avec la base de données.

------------------------------------------------------------------------

## 2. Architecture

``` text
Navigateur / Mobile
          │
          ▼
      API REST
          │
          ▼
 Services Métier
          │
          ▼
 Repositories SQL
          │
          ▼
      Base SQL
```

------------------------------------------------------------------------

## 3. Principes

L'API respecte les principes suivants :

-   REST stateless ;
-   échanges JSON UTF-8 ;
-   versionnement explicite ;
-   validation systématique des entrées ;
-   documentation OpenAPI.

------------------------------------------------------------------------

## 4. Préfixe des routes

Toutes les routes utilisent le préfixe :

``` text
/api/v1
```

Une future version incompatible utilisera :

``` text
/api/v2
```

Cette règle applique la politique de compatibilité ascendante définie
en L007 §7.1 : tout changement cassant (suppression/renommage de champ,
changement de sémantique d'un code HTTP) requiert une nouvelle version.

## 6. Format des réponses

Toutes les réponses suivent le même modèle.

``` json
{
  "success": true,
  "timestamp": "2026-01-01T12:00:00Z",
  "version": "1.0",
  "data": {}
}
```

En cas d'erreur :

``` json
{
  "success": false,
  "error": {
    "code": "COURSE_NOT_FOUND",
    "message": "Course introuvable"
  }
}
```

Le champ `code` correspond à un identifiant stable dérivé de la
hiérarchie d'exceptions applicatives (cf. L023 §4), permettant à un
client de distinguer par programmation les cas d'erreur sans analyser
le texte du message.

------------------------------------------------------------------------

## 7. Codes HTTP

  Code   Utilisation
  ------ -------------------
  200    Succès
  201    Création
  204    Aucun contenu
  400    Requête invalide
  401    Non authentifié
  403    Accès refusé
  404    Ressource absente
  409    Conflit
  422    Validation métier
  500    Erreur interne

------------------------------------------------------------------------

## 8. Pagination

Les listes volumineuses utilisent :

-   `page`
-   `size`
-   `sort`

Réponse :

``` json
{
  "page":1,
  "size":50,
  "total":250,
  "items":[]
}
```

------------------------------------------------------------------------

## 9. Principes de développement

-   aucune logique métier dans les routes ;
-   validation Pydantic ;
-   services injectés ;
-   journalisation systématique ;
-   documentation OpenAPI générée automatiquement.
