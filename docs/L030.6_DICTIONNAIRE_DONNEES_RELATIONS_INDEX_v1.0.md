## 1. Objet

Ce document clôt le dictionnaire de données de TurfIA en définissant les
relations entre les tables, les contraintes d'intégrité et la stratégie
d'indexation.

------------------------------------------------------------------------

# 4. Clés primaires

Toutes les tables utilisent une clé primaire technique :

``` text
id BIGINT
```

Les identifiants sont générés automatiquement.

------------------------------------------------------------------------

# 5. Clés étrangères

Les clés étrangères suivent la convention :

``` text
id_<table>
```

Exemples :

-   hippodrome_id
-   reunion_id
-   course_id
-   cheval_id
-   analyse_id

Toutes les relations sont protégées par des contraintes `FOREIGN KEY`.

------------------------------------------------------------------------

# 6. Contraintes UNIQUE

Les principales contraintes d'unicité sont :

  Table       Colonnes
  ----------- -----------------------------
  reunion     date, hippodrome_id, numero
  course      reunion_id, numero
  partant     course_id, numero
  partant     course_id, cheval_id
  analyse     course_id, version
  parametre   cle
  version     version

Ces contraintes empêchent la création de doublons.

------------------------------------------------------------------------

# 7. Contraintes CHECK

Les contrôles suivants sont recommandés :

  Champ             Règle
  ----------------- -----------------
  score_confiance   0 ≤ score ≤ 100
  mise              ≥ 0
  gains             ≥ 0
  roi               libre
  numero            \> 0
  classement        \> 0

Les contraintes métier complexes restent implémentées dans la couche
applicative.

------------------------------------------------------------------------

# 8. Politique de suppression

Les suppressions physiques sont interdites pour les données métier.

La politique retenue est la suivante :

-   référentiels : désactivation logique ;
-   courses : conservation permanente ;
-   analyses : conservation permanente ;
-   statistiques : conservation permanente ;
-   journaux : archivage selon politique de rétention.

------------------------------------------------------------------------

# 9. Stratégie d'indexation

## Index simples

Création systématique sur :

-   date
-   nom
-   score_confiance
-   roi
-   version
-   numero

------------------------------------------------------------------------

## Index composites

  Table      Colonnes
  ---------- -----------------------
  reunion    date, hippodrome_id
  course     reunion_id, numero
  partant    course_id, numero
  cote       partant_id, date_maj
  analyse    course_id, version
  resultat   course_id, classement

------------------------------------------------------------------------

# 10. Optimisation SQL

Les requêtes fréquentes utilisent :

-   les index de recherche ;
-   les vues SQL ;
-   des jointures limitées ;
-   la pagination.

Les scans complets sont évités autant que possible.

------------------------------------------------------------------------

# 11. Intégrité référentielle

Toutes les écritures respectent les règles suivantes :

-   aucun enregistrement orphelin ;
-   aucune clé étrangère invalide ;
-   transactions atomiques ;
-   rollback en cas d'échec.

------------------------------------------------------------------------

# 12. Historisation

Le modèle interdit la perte d'information.

Les principes sont :

-   aucune analyse supprimée ;
-   aucune statistique supprimée ;
-   aucune cote remplacée ;
-   toutes les versions conservées.

Cette historisation permet la reproductibilité intégrale des analyses.

------------------------------------------------------------------------

# 13. Évolutivité

Le schéma est conçu pour intégrer sans refonte :

-   de nouvelles disciplines ;
-   de nouveaux types de paris ;
-   de nouveaux algorithmes ;
-   plusieurs versions simultanées du modèle TurfIA ;
-   de nouveaux indicateurs statistiques.

------------------------------------------------------------------------

# 14. Conclusion

Le dictionnaire de données L030 constitue la spécification officielle de
la base relationnelle de TurfIA.

Il sert de référence pour :

-   les scripts SQL (`CREATE TABLE`) ;
-   les migrations ;
-   les modèles SQLAlchemy ;
-   les schémas Pydantic ;
-   les routes FastAPI ;
-   les traitements statistiques.

Toute évolution du modèle devra être cohérente avec cette spécification.
